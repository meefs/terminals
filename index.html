Drawing in the Terminal, 2026

What replaced libcaca — and the quiet graphics renaissance that came with it.

COMPILED BY CLAUDE (OPUS 4.8)  ·  2026-06-18  ·  SOURCES: GITHUB + NPM (LIVE)

The one-line answer

The modern libcaca is chafa.

But the bigger truth is that the terminal stopped pretending. The frontier moved from "approximate a picture with characters" to real pixels over a wire protocol (Kitty / Sixel), with fancy Unicode glyphs as the graceful fallback. chafa wins precisely because it spans the whole ladder. libcaca / aalib are now history exhibits, not tools to build on.

1

The lineage

aalib (1997, monochrome ASCII) → libcaca (2002, "Colour ASCII Art" + a cacaview/img2txt CLI) defined the genre: take an image, map cells to characters by luminance, dither in 16 colors. Brilliant for its era — and effectively frozen since. libcaca's last release is v0.99.beta20 from 2021; the repo's own description is a literal 💩 emoji. It still ships in distros, but nobody starts new work on it.

What changed underneath it: 24-bit truecolor went mainstream, Unicode kept minting denser block glyphs, GPUs got into terminals, and three image protocols matured. The successors split along those lines.

3

The two things that actually changed

A · Graphics protocols — the real "modern rendering"

Terminals can now stream actual raster images into the grid. Three protocols, in rough order of momentum (hover a card & click to read the spec):

↗

Kitty Graphics

The emerging standard. PNG/animation, placement, z-index. Supported by kitty, WezTerm, Ghostty, Konsole. Target this first.

 ↗

Sixel

1980s DEC format, resurgent. The broad-compat fallback: xterm, foot, mlterm, WezTerm, Konsole, Windows Terminal.

 ↗

iTerm2 inline

Base64-an-image escape. macOS/iTerm2 + WezTerm. Simple; narrower reach.

You rarely implement these by hand — you let chafa, timg, terminal-image or a TUI lib probe the terminal and pick. To actually see pixels you need a capable emulator (see §6).

B · The character-cell resolution ladder kept climbing

When there's no protocol (SSH to a basic box, CI logs, a retro vibe), you fake pixels with block glyphs. Unicode keeps adding denser ones — each step doubles the "sub-pixels" per character cell:

Glyph setSub-cellsSinceNote
ASCII (luminance ramp)1 (brightness)1963the @%#*+=-:. aesthetic
Half blocks ▀2×1legacyfg+bg = 2 color pixels/cell
Quadrants ▟2×2legacy4 sub-cells
Sextants2×3Unicode 13 (2020)"Symbols for Legacy Computing"
Octants newest2×4Unicode 16 (2024)8 sub-cells — chafa supports them
Braille ⠿2×4 dotsUnicode (Braille)monochrome but dense — great for line art / plots

The edge today: chafa's support for octants (2×4, Unicode 16) is the genuinely-latest move in pure-character rendering — finer grayscale than the sextants everyone copied in 2021. For monochrome density (and terminal plots), braille remains the portable trick.

The enablers: terminals that can actually draw

None of the pixel stuff matters if the emulator can't render it. Your renderer is only as good as the terminal under it (hover & click each):

↗

Ghostty momentum

Native Metal on macOS → 120Hz/ProMotion, adaptive sync, power-efficient. Kitty protocol. The 2025–26 darling.

 ↗

kitty

GPU-accelerated; origin of the Kitty graphics protocol. The reference implementation.

 ↗

WezTerm

The completist: the only terminal that speaks all three image protocols (Kitty + Sixel + iTerm2).

 ↗

Konsole

KDE's workhorse — Kitty + Sixel. Mature, configurable, a solid Linux default.

 ↗

foot

Lean, fast Wayland-native terminal with Sixel. The minimalist Linux pick.

 ↗

Apple Terminal avoid

No image protocol, and still no truecolor in 2026. The lowest common denominator — test your fallbacks here.
7

My take & guidance

If you just want to throw an image in the terminal

# chafa auto-detects kitty/sixel/iterm2 and falls back to unicode. that's it.
chafa image.png
chafa -f sixel movie.gif            # force a protocol
chafa --symbols octant photo.jpg    # force fanciest char art (Unicode 16)

One tool, every output mode, actively developed. Don't overthink it — chafa it is.

If you're embedding in an app

Link libchafa (C/FFI) for image→cells. For a whole UI, use your language's framework — and let it own rendering rather than hand-rolling escape codes:

go get github.com/charmbracelet/bubbletea   # Go
cargo add ratatui                            # Rust
pip install textual                          # Python
npm i ink react                              # Node
npm i terminal-image                         # Node, images only

Where the puck is going: bet on the Kitty graphics protocol as primary, Sixel as fallback, octant/half-block Unicode as the floor. Ghostty's GPU-native rise is dragging "real graphics in the terminal" from novelty to expectation. The good news: chafa / timg / terminal-image already encode that exact ladder, so you usually inherit it for free.

The potholes nobody warns you about:  (1) tmux & SSH swallow image protocols unless you enable passthrough.  (2) Cell aspect ratio isn't 1:1 (~1:2 tall) — naive rendering squashes images.  (3) Terminal detection is a swamp — never assume; probe (or lean on a lib).  (4) Scrollback + images interact weirdly.  (5) Test on Apple Terminal to see your true fallback.

The verdict in one breath

chafa is the modern libcaca — full stop. But "terminal rendering in 2026" is a much bigger story: GPU terminals + Kitty/Sixel protocols for real pixels, the octant/braille Unicode ceiling for fallback, and a per-language TUI-framework boom (Bubble Tea / Ratatui / Textual / Ink) where most actual "drawing" now happens. libcaca and aalib walked so this could run. 🫡
